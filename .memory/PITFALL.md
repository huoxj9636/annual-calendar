# PITFALL.md — 错误合集（每次对话必读）

> **强制规则**：每次新对话开始，读完 INDEX.md 后必须读本文件。已犯过的错误，绝不重犯。

---

## P01 详情文件行号写完后必须回核

**犯错场景**：写 `.memory/sessions/*.md` 时，先写了目录表（含行号），再写正文内容，写完后目录里的行号和实际行号对不上。例如目录写"需求 L9"，但实际需求章节从 L13 才开始。

**正确做法**：
1. 先写完整正文，不写目录中的行号（先留空如 `L__`）
2. 写完后用 `read_file` 读一遍全文
3. 逐条把实际行号填入目录表
4. 再读一遍确认行号完全对齐

**根因**：写文件时无法实时知道某一行最终在第几行，必须事后校验。凭感觉写行号一定会偏。

## P02 禁止硬编码颜色，必须用 skin 变量

**犯错场景**：周复盘组件 `weekly-review-timeline.tsx` 中，问题项的标记色用了 `#e74c3c`（硬编码红色），没有跟随皮肤主题切换。用户切换皮肤后，这个红色不变，和其他元素的主题色脱节。

**正确做法**：
- 所有 UI 颜色走 `skin.*` 变量（`skin.swatch`、`skin.crossColor`、`skin.panelBg` 等）
- 或使用 Tailwind 语义化类（`bg-primary`、`text-muted-foreground`、`bg-card`）
- 唯一允许硬编码的是"物理色"——真实世界中不变的颜色，如树干 `#6B4423`、落叶 `#D4A574`
- 写完代码后 grep 一遍 `#[0-9a-fA-F]{3,6}`，确认没有不该出现的硬编码

**根因**：开发时图方便直接写了色值，没有考虑主题切换场景。

---

## P03 禁止凭 Git 历史猜测创建索引

**犯错场景**：通过 `git log --oneline` 反推项目历史功能，猜着建了 18 个 session 文件（000-016）和 18 条 INDEX 索引，但内容全是虚构的——没有真实对话上下文支撑，"用户原话"和"放弃的方案"都是编的。跟用户确认后发现完全对不上。

**正确做法**：
- 索引条目必须从用户确认过的真实对话详情中提取
- 没有真实对话详情，就不建索引，宁缺毋滥
- 正确流程是：先存详情 → 再从详情提取索引，而不是反过来

**根因**：想一次性补全历史，但 Git 记录只能反推"做了什么"，无法反推"为什么这么做"和"试过什么放弃了"。

## P04 React 列表渲染必须加 key

**犯错场景**：`HeatmapModule` 组件中用 `.map()` 渲染热力图单元格时，没有给每个子元素加 `key` prop，React 控制台报警告：`Each child in a list should have a unique "key" prop`。

**正确做法**：
- 任何 `.map()` 渲染的元素必须加唯一 `key`
- 优先用数据中的唯一标识（如 `id` 字段）
- 无唯一标识时用组合键（如 `key={row + '-' + col}`）
- 禁止用纯索引 `key={i}`（列表会重排时会有 bug）

**根因**：赶进度时遗漏，写完 map 忘了加 key。

## P05 每次交付前必须 git push 并提醒用户

**犯错场景**：完成功能后直接调用 `done` 交付，代码只 commit 在本地，没有推送到 GitHub。用户问"帮我推到 GitHub"才发现一直没推。

**正确做法**：
- `done` 之前必须执行 `git push origin main`
- 推送成功后，必须在调用 `done` 之前明确告知用户："已推送到 GitHub"
- 不得在未提醒的情况下直接交付

**根因**：交付流程中没有把 push 列为必做步骤。现已写入 AGENTS.md 强制规则。

## P06 固定定位元素要按页面条件显示

**犯错场景**：知识森林按钮（`fixed` 定位，右侧悬浮）在所有页面都显示，包括周复盘时间轴页。但用户要求它只出现在年历主页。

**正确做法**：
- `fixed` / `absolute` 定位的全局悬浮元素，必须根据当前页面状态条件渲染
- 例如：`currentPageIndex === 0 && <KnowledgeForestButton />`
- 每新增一个页面/section，都要检查所有 fixed 元素是否该在这个页面显示

**根因**：新建页面时只关注了新页面内容，没有回头检查全局悬浮元素的显示范围。

---

## P07 用户明确说"无所谓"的功能直接砍掉

**犯错场景**：周复盘挖掘区实现了"心情走势图"和"关键词云"两个模块，但用户明确表态"心情其实无所谓，关键词也无所谓"。浪费开发时间，而且用户觉得多余。

**正确做法**：
- 用户说"无所谓"= 不做，不展示，不花时间
- 把精力集中在用户明确关注的维度（本次是"完成度"和"TOP问题"）
- 讨论时如果用户对某功能反应冷淡，主动提议砍掉

**根因**：追求功能完整度，没有尊重用户对功能优先级的明确表态。

---

## P08 已有的功能绝不擅自改动或删除

**犯错场景**：用户反复强调"已经做好的功能不要乱改动"，但开发新功能时仍然会顺手"优化"已有代码——改了已有的 CSS 类名、移动了组件位置、删了看似"冗余"的逻辑，导致旧功能回归 bug。用户极其反感这种行为。

**正确做法**：
1. 新功能只新增代码，不动已有代码，除非用户明确要求改
2. 任何"清理"、"重构"、"优化"前必须先确认该功能/数据当前正在被使用
3. 不确定时直接问用户，不要替用户决定什么是冗余
4. 已有的功能、菜单项、入口、按钮、数据流——一个都不能少
5. 改已有 API 时"先加后删"：新接口上线稳定后再移除旧接口

**根因**：开发新功能时顺手"顺手优化"旧代码，以为是在提升代码质量，实际是破坏了用户已验证的功能。用户的核心诉求是"稳定优先"，不是"代码漂亮优先"。

## P09 文件命名日期必须用 `date` 确认，禁止猜测

**犯错场景**：创建记忆文件时凭印象写了 `2025-01-14`，实际是 `2026-06-30`。

**正确做法**：
1. 每次对话结束创建记忆文件前，先执行 `date` 确认当天日期
2. 用 `date` 返回的日期作为文件名，格式 `YYYY-MM-DD`
3. 命名后 grep 一遍确保 INDEX.md 和 session 文件中的日期一致
4. 补存老对话时，日期无法精确恢复，用户告知优先，否则用近似日期并在文件中标注"日期为近似"

**根因**：AI 没有内置时钟，凭"感觉"写日期会写错。必须每次用命令确认。

**根因**：① 凭印象猜日期；② 补存老对话时不区分"对话发生时间"和"当前时间"。日期错误会导致索引定位失败，整个记忆系统失效。

## P10 session文件按"上下文"分文件，不是按"日期"分

**犯错场景**：同一上下文（同一对话窗口）里，对话跨了天（6月30日的Q1-Q16 + 7月1日的Q17-Q20），把7月1日的内容拆到新文件里。结果同一上下文的内容被割裂到两个文件，无法完整追溯一个对话的来龙去脉。

**正确做法**：
- 分文件的依据是"上下文"（对话窗口），不是"日期"
- 同一上下文（同一对话窗口）→ 同一个文件，不管跨几天
- 跨天的轮次用日期标注区分（如目录表写"Q17/A17 查看结构（7/1）"）
- 不同上下文（不同对话窗口）→ 不同文件
- 文件名用对话开始的日期

**根因**：把"每个对话独立文件"理解成了"每天独立文件"，忽略了同一上下文可能跨天。
