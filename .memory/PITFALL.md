# PITFALL.md — 错误合集（每次对话必读）

> **强制规则**：每次新对话开始，读完 INDEX.md 后必须读本文件。已犯过的错误，绝不重犯。

---

## P01 详情文件行号写完后必须回核

**犯错场景**：写 `.memory/sessions/*.md` 时，先写了目录表（含行号），再写正文内容，写完后目录里的行号和实际行号对不上。例如目录写"需求 L9"，但实际需求章节从 L13 才开始。

**正确做法**：
1. 先写完整正文，不写目录中的行号（先留空如 `L__`）
2. 写完后用 `read_file` 读一遍全文
3. 逐条把实际行号填入目录表
4. 再读一遍确认行号完全对齐

**根因**：写文件时无法实时知道某一行最终在第几行，必须事后校验。凭感觉写行号一定会偏。

## P02 禁止硬编码颜色，必须用 skin 变量

**犯错场景**：周复盘组件 `weekly-review-timeline.tsx` 中，问题项的标记色用了 `#e74c3c`（硬编码红色），没有跟随皮肤主题切换。用户切换皮肤后，这个红色不变，和其他元素的主题色脱节。

**正确做法**：
- 所有 UI 颜色走 `skin.*` 变量（`skin.swatch`、`skin.crossColor`、`skin.panelBg` 等）
- 或使用 Tailwind 语义化类（`bg-primary`、`text-muted-foreground`、`bg-card`）
- 唯一允许硬编码的是"物理色"——真实世界中不变的颜色，如树干 `#6B4423`、落叶 `#D4A574`
- 写完代码后 grep 一遍 `#[0-9a-fA-F]{3,6}`，确认没有不该出现的硬编码

**根因**：开发时图方便直接写了色值，没有考虑主题切换场景。

## P03 禁止凭 Git 历史猜测创建索引

**犯错场景**：通过 `git log --oneline` 反推项目历史功能，猜着建了 18 个 session 文件（000-016）和 18 条 INDEX 索引，但内容全是虚构的——没有真实对话上下文支撑，"用户原话"和"放弃的方案"都是编的。跟用户确认后发现完全对不上。

**正确做法**：
- 索引条目必须从用户确认过的真实对话详情中提取
- 没有真实对话详情，就不建索引，宁缺毋滥
- 正确流程是：先存详情 → 再从详情提取索引，而不是反过来

**根因**：想一次性补全历史，但 Git 记录只能反推"做了什么"，无法反推"为什么这么做"和"试过什么放弃了"。

## P04 React 列表渲染必须加 key

**犯错场景**：`HeatmapModule` 组件中用 `.map()` 渲染热力图单元格时，没有给每个子元素加 `key` prop，React 控制台报警告：`Each child in a list should have a unique "key" prop`。

**正确做法**：
- 任何 `.map()` 渲染的元素必须加唯一 `key`
- 优先用数据中的唯一标识（如 `id` 字段）
- 无唯一标识时用组合键（如 `key={row + '-' + col}`）
- 禁止用纯索引 `key={i}`（列表会重排时会有 bug）

**根因**：赶进度时遗漏，写完 map 忘了加 key。

## P05 每次交付前必须 git push 并提醒用户

**犯错场景**：完成功能后直接调用 `done` 交付，代码只 commit 在本地，没有推送到 GitHub。用户问"帮我推到 GitHub"才发现一直没推。

**正确做法**：
- `done` 之前必须执行 `git push origin main`
- 推送成功后，必须在调用 `done` 之前明确告知用户："已推送到 GitHub"
- 不得在未提醒的情况下直接交付

**根因**：交付流程中没有把 push 列为必做步骤。现已写入 AGENTS.md 强制规则。

## P06 固定定位元素要按页面条件显示

**犯错场景**：知识森林按钮（`fixed` 定位，右侧悬浮）在所有页面都显示，包括周复盘时间轴页。但用户要求它只出现在年历主页。

**正确做法**：
- `fixed` / `absolute` 定位的全局悬浮元素，必须根据当前页面状态条件渲染
- 例如：`currentPageIndex === 0 && <KnowledgeForestButton />`
- 每新增一个页面/section，都要检查所有 fixed 元素是否该在这个页面显示

**根因**：新建页面时只关注了新页面内容，没有回头检查全局悬浮元素的显示范围。

## P07 用户明确说"无所谓"的功能直接砍掉

**犯错场景**：周复盘挖掘区实现了"心情走势图"和"关键词云"两个模块，但用户明确表态"心情其实无所谓，关键词也无所谓"。浪费开发时间，而且用户觉得多余。

**正确做法**：
- 用户说"无所谓"= 不做，不展示，不花时间
- 把精力集中在用户明确关注的维度（本次是"完成度"和"TOP问题"）
- 讨论时如果用户对某功能反应冷淡，主动提议砍掉

**根因**：追求功能完整度，没有尊重用户对功能优先级的明确表态。

## P08 已有的功能绝不擅自改动或删除

**犯错场景**：用户反复强调"已经做好的功能不要乱改动"，但开发新功能时仍然会顺手"优化"已有代码——改了已有的 CSS 类名、移动了组件位置、删了看似"冗余"的逻辑，导致旧功能回归 bug。用户极其反感这种行为。

**正确做法**：
1. 新功能只新增代码，不动已有代码，除非用户明确要求改
2. 任何"清理"、"重构"、"优化"前必须先确认该功能/数据当前正在被使用
3. 不确定时直接问用户，不要替用户决定什么是冗余
4. 已有的功能、菜单项、入口、按钮、数据流——一个都不能少
5. 改已有 API 时"先加后删"：新接口上线稳定后再移除旧接口

**根因**：开发新功能时顺手"顺手优化"旧代码，以为是在提升代码质量，实际是破坏了用户已验证的功能。用户的核心诉求是"稳定优先"，不是"代码漂亮优先"。

## P09 文件命名日期必须用 `date` 确认，禁止猜测

**犯错场景**：创建记忆文件时凭印象写了 `2025-01-14`，实际是 `2026-06-30`。

**正确做法**：
1. 每次对话结束创建记忆文件前，先执行 `date` 确认当天日期
2. 用 `date` 返回的日期作为文件名，格式 `YYYY-MM-DD`
3. 命名后 grep 一遍确保 INDEX.md 和 session 文件中的日期一致
4. 补存老对话时，日期无法精确恢复，用户告知优先，否则用近似日期并在文件中标注"日期为近似"

**根因**：AI 没有内置时钟，凭"感觉"写日期会写错。必须每次用命令确认。

**根因**：① 凭印象猜日期；② 补存老对话时不区分"对话发生时间"和"当前时间"。日期错误会导致索引定位失败，整个记忆系统失效。

## P10 session文件：同一上下文同一文件，禁止翻目录找别人的文件

**犯错场景**：Agent 在新对话里，主动扫描 .memory/sessions/ 目录，看到同日期的旧文件就把内容追加进去了。但那是别的对话窗口的文件，不应该追加。

**正确做法**：
- 同一个上下文 = 同一个文件（追加）
- 不同上下文 = 不同文件（新建）
- 只看当前上下文自己的对话记录判断有没有创建过文件，不翻目录
- 禁止扫描 sessions 目录、禁止用日期匹配旧文件

**根因**：Agent 主动翻目录去找文件追加，但目录里的文件是别的对话窗口的。

## P11 目录表每条记录都必须标注日期

**犯错场景**：session详情文件的目录表只写"编号/主题/行号"三列，日期只在跨天时用括号标注在主题后面（如"查看结构（7/1）"），同一天的记录没有日期。结果读目录时无法知道每轮对话是哪天的，只有跨天的才有标注。

**正确做法**：
- 目录表增加独立的"日期"列，每条记录都标注（如 6/30、7/1）
- 格式：`| 编号 | 日期 | 主题 | 行号 |`
- 不允许只在跨天时才标注日期，同一天的记录也要写日期

**根因**：图省事只在跨天时标注，没考虑"每条记录都该有日期"这个基本需求。

## P12 API 写入操作必须传 NOT NULL 字段，且后端错误不能静默吞掉

**犯错场景**：OKR 的 POST `/api/okr` 路由中，3 张表都有 `user_id NOT NULL` 约束，但 upsert 时从未传入 `user_id`，导致所有写入必然触发约束违反。后端只 `console.error` 仍返回 `{ success: true }`，前端 `.catch(() => {})` 静默吞错。用户新增了 5 个 OKR，全部只存在于内存中，新版本部署后全部丢失，用户完全无感知。

**正确做法**：
1. 写 upsert 时必须检查目标表的所有 NOT NULL 字段是否都有值，缺任何一个都会静默失败
2. 后端遇到 upsert 错误应收集到数组，有错误时返回 500 + 错误详情，不能仍返回 success:true
3. 前端不能 `.catch(() => {})` 吞错，至少要检查 res.ok，失败时给用户可见的错误提示
4. 新增 API 路由时，先用 `exec_sql` 查一遍目标表的 schema（column_name / is_nullable / column_default），确认哪些字段必填

**根因**：① 开发时没查数据库 schema，不知道 user_id 是 NOT NULL；② 错误处理图省事——后端"打 log 就行"、前端"吞掉就行"，导致问题完全被掩盖。

## P13（7/1）测试 API 用真实年份污染用户数据

**犯错场景**：用 `curl -d '{"type":"drawing","year":2026,...}'` 测试 calendar-data API，把用户的画布数据覆盖成脏数据 `[{x:10,y:20,color:"#ff0000"}]`。后续 `SELECT * FROM calendar_drawings WHERE year=2026` 显示 4 天前就空，**但污染行为本身仍不可接受**——用户原数据可能存在其他位置（localStorage），DB 数据被污染后即使主源还在也增加了恢复复杂度。

**正确做法**：
- **所有 API 测试必须用沙箱年份 `year=2099` 或 `9999`**，绝对禁止用 2025/2026 等真实年份
- 测试 POST body 也要带 `user_id: 'test-isolated'` 隔离
- 任何"读+写"的测试操作前先 SELECT 确认现状，写完立刻还原或 DELETE
- 写代码时遇到 user_id NOT NULL 约束：**不要写假数据绕过**，要么加占位 `'legacy'`，要么问用户

**根因**：没意识到测试操作对真实数据有副作用，违反了"测试/调试严禁污染真实数据"的红线规则（已写入 AGENTS.md 第 2 条）。

## P14（7/1）localStorage 迁移时 removeItem 导致数据丢失风险

**犯错场景**：drawing-overlay.tsx 旧代码逻辑是"DB 无数据 → 从 localStorage 读取 → 同步到 DB → `localStorage.removeItem(storageKey)` 删除本地"。如果 DB 同步失败（网络/服务挂了）或后续 DB 损坏，**localStorage 已删 + DB 无数据 = 数据永久丢失**。

**正确做法**：
- **localStorage 永远不删**（永久作为本地备份）
- 同步 DB 是"追加"行为，不是"迁移"
- 加载时优先取 localStorage，同步到 DB（不删 localStorage）
- 如果非要"删除 localStorage"，必须满足两个条件：DB 同步成功 + 服务端 200 OK + 本地有审计日志
- 加 3 重备份：主 key + 备份 key + DB

**根因**：把 localStorage 当成"临时缓存"而非"本地真相"。它应该是**主存储**（用户最近操作），DB 是**云备份**。这条违反"用户数据永远不删"的红线规则（已写入 AGENTS.md 第 1 条）。

## P15（7/1）后端返回格式与前端消费不一致的隐患

**犯错场景**：后端 calendar-data GET drawing 返回 `data?.strokes || null`（**裸数组**），前端 `data.strokes && Array.isArray(data.strokes)` 期望 `{strokes: [...]}` **包装对象**。后端 `data.strokes` 在裸数组上访问不到，**永远 undefined**，前端走 localStorage 兜底分支（而 localStorage 可能也没有）。整条数据加载链路静默失败。

**正确做法**：
- API 返回结构**必须统一包装**：`{strokes: [...]}` 而不是裸数组
- 或：前端读取时**两种格式都兼容** `Array.isArray(data) ? data : (data?.strokes || [])`
- **接口契约**（后端返回结构）应该用 TypeScript 类型导出共享，前后端共用：
  ```ts
  // src/lib/api-types.ts
  export type GetDrawingResponse = { strokes: Stroke[] } | { error: string };
  ```
- 静默失败**不可接受**：如果后端返回结构不对，前端应该 throw console.error 而不是"看起来正常但实际不工作"

**根因**：后端 API 设计没考虑前端消费契约，导致接口"看起来工作"但实际上读不到数据。这类 bug 用户体验上是"画布数据莫名其妙消失"。

## P16（7/1）新功能方案讨论太久不落地导致用户暴怒

**犯错场景**：用户多次反馈"画布不见了"、"数据显示不出来"，AI 一直在"解释原因 + 给方案 + 询问要不要做"，用户被逼到骂人"妈了个逼的。现在能不能给我解决这个问题啊？"。

**正确做法**：
- **用户说"解决"时不要再问方案**，直接做兜底（备份/恢复/降级）
- 3 步以内能做的兜底（加备份层、改逻辑、加日志）**不要问、直接做**
- 只有"业务决策"才问（登录方式、数据合并方式等不可逆的）
- **承认错误要快**：用户骂时不要解释，第一句"是我的错"+ 第二句"我立即做 X 兜底"
- 永远不让用户做"诊断操作"（查 localStorage、看 console）—— **AI 自己想办法**

**根因**：把用户当协作方（讨论方案），实际用户要的是**问题解决者**（直接修好）。修复类对话的 SLA 是"修好"，不是"达成共识"。

## P17（7/1）edit_file 多次替换同一文件时破坏代码结构

**犯错场景**：在 `year-calendar.tsx` 中连续用 edit_file 做多处替换，其中一处替换插入的代码块破坏了已有的 try-catch 结构（catch 块被吞掉），导致 `Parsing error: 'catch' or 'finally'` 报错，tsc 5 个错误。必须回读文件修复结构。

**正确做法**：
1. 对同一文件做多处 edit_file 替换时，每次替换后必须 `read_file` 回读修改区域，确认结构完整
2. 特别注意 try-catch-finally、if-else、花括号配对——替换 old_string 时如果包含了某个右花括号，new_string 必须也包含它
3. 连续多处替换后，跑一遍 `test_run` 或 `tsc --noEmit`，趁早发现结构破坏
4. 如果文件很长，替换前先 `read_file` 精准定位 old_string 的上下文，不要凭记忆写

**根因**：多处修改同一文件时，每次只关注当前替换点，没有检查替换是否破坏了周围代码的语法结构。

## P18（7/1）记忆整理偷懒：只记几轮，遗漏大部分对话

**犯错场景**：用户要求整理整个上下文的对话到记忆系统。AI第一次只记了3轮（Q1-Q3），被抓后补到5轮。但整个会话（包括上下文耗尽前的Phase A-L）实际有20+轮对话，前上下文的12个Phase完全没记录。

**正确做法**：
1. 整理记忆时，必须从对话摘要+当前上下文完整梳理每一轮用户-AI交互
2. 如果有"上下文耗尽前的摘要"，摘要中的每个Phase都是一轮对话，必须全部记录
3. 不要只看当前上下文的用户消息，要回溯到摘要中列出的所有Phase
4. 记忆的价值在于完整性——遗漏的对话 = 丢失的决策历史 = 下次对话重复犯错

**根因**：图省事，只看当前上下文可见的用户消息，没有认真梳理摘要中的历史对话。对"完整对话"的定义太窄。

## P19（7/11）修改格子长度/时间计算时破坏了时间块创建功能

**犯错场景**：在甘特图功能中，反复修改 cellWidth/scale/totalSlots 计算方式以解决格子长度问题。每次修改后，之前正确的时间块创建位置就错了——在晚上8点新增的时间块不在8点往后加半小时，而是跑到前面去了。用户愤怒骂"你他妈的"。

**正确做法**：
1. 凡涉及"时间计算"的修改（cellWidth/scale/hoverHour/创建位置），修改后**必须立即测试**所有与时间计算相关的功能：创建时间块、拖拽移动、拖拽调整长度、悬浮时间线显示
2. 创建测试用例：分别在 0点、12点、18点、23点 创建时间块，确认位置都正确
3. 核心公式变动后，检查所有使用到 cellWidth 和 scale 的地方：hoverHour、创建位置、时间线位置、时间块宽度
4. 改用 `test_run` 做简单验证，不要依赖肉眼检查

**根因**：时间计算是一个耦合系统（cellWidth/scale/hoverHour/创建位置/渲染宽度），改一处时必须同步检查所有消费方。只验证了"格子长度变对了"就以为改好了，忘了验证"创建位置"。

## P20（7/11）23:45空白区问题反复修改三次没改对

**犯错场景**：用户要求在30分/15分钟档消除23:45后的空白区域。第一次改了右内边距（视觉上少了点空白但格子还在），第二次改格子但理解错方向，第三次才彻底搞对。用户说"这个问题我说了三遍，你每次都说改了，但每一次都没有改好"。

**正确做法**：
1. 先确认用户说的"空白"是格子本身太长还是格子后面的多余空间
2. 确认后立即在代码中定位到**具体哪一行**控制这个长度
3. 修改后用 read_file 确认修改生效，刷新页面验证
4. 如果一次验证不通过，立即重新理解问题，不要重复提交同样的无效修改

## P21（7/12）模块顺序加载过滤只认内建key，自定义书签被丢

**犯错场景**：年历主页"更多"弹窗里添加的自定义链接，刷新页面后从左侧列表里消失。根因：`useEffect` 加载 `calendar-module-order` 时，用 `Object.keys(defaultModuleNames).includes(k)` 过滤合法 key，这个集合只包含 8 个内建模块 ID（timeline/dida/.../achievement），自定义书签的 ID（如 `bm_1234567890`）被当作"非法值"过滤掉，于是 `moduleOrder` 不包含书签，渲染时 `moduleOrder.filter(...)` 自然不显示。

**正确做法**：
1. 任何"白名单过滤"涉及动态新增数据时，白名单必须包含**所有可能的数据类型**（内建 + 自定义），不能只覆盖初始静态集合
2. 加载时如果 localStorage 里的 ID 不在白名单，**先问自己**：这个 ID 是历史脏数据，还是新功能的合法值？
3. 对"自定义数据"和"内建数据"混存的字段，校验白名单要同时枚举：`new Set([...builtinIds, ...customIds])`
4. 修完后用 test_run 走 lint + ts-check，再用浏览器手动走一遍「添加→刷新→还在」全链路
5. 注意 `moduleVisibility` 用的是 spread 合并（`{...prev, ...saved}`），所以那个字段天然兼容，但 `moduleOrder` 用的是 filter 重建，所以丢了——两种"加载模式"行为不一致，调试时要分别看

**根因**：① 写白名单时只考虑"当前数据范围"，没有考虑"功能扩展后会新增的 ID 类型"；② 没意识到"加载"和"保存"用了不同策略（保存是 push 追加，加载是 filter 重建），导致不对称。

**根因**：① 没有仔细区分"格子长度"和"格子后面多余的空白"是两个不同的概念；② 修改后没有在真实页面中验证就认为改好了；③ 连续三次没有真正理解问题的本质，导致用户极其不满。
