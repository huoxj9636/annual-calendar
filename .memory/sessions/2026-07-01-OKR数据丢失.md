## Q1（7/1）

**用户**：就是人生旅途 OKR 那个页面啊，我去新增 OKR 我新增了5个，就是 O1、 O2、 O3、 O4、 O5。但是我新版本覆盖的时候，O3到 O4的数据全不见了。我不知道是不是没有存到数据库，还说整个就没打通，你查一下呢。是不是整个没打通？还是什么其他原因？

**AI**：

查看了 `life-calendar.tsx` 中 OKR 数据模型（OKRObjective → OKRKeyResult → OKRTask 三层结构），发现数据流：
- 加载：`GET /api/okr` → 从 Supabase 读
- 保存：`useEffect` 监听 `goals` 变化 → `POST /api/okr` 写回

检查 `POST /api/okr` 的 route.ts，发现 **根因链**：

1. 数据库 3 张 OKR 表（okr_objectives / okr_key_results / okr_tasks）都有 `user_id NOT NULL` 约束，无默认值
2. POST 的 upsert 从未传入 `user_id` → 每次写入必然触发 NOT NULL 约束违反
3. 后端只 `console.error`，仍然返回 `{ success: true }`
4. 前端 `.catch(() => {})` 静默吞掉所有错误

结论：**OKR 数据从来没有真正写入过数据库**，5 个 OKR 全部只存在于浏览器内存中，刷新/部署后全部丢失。

另外发现 GET 端点也没有按 user_id 过滤，会返回所有用户的混合数据；数据库 RLS 策略允许 `user_id = 'legacy'` 的记录被读取。

**修复内容**：

| 位置 | 修复 |
|------|------|
| `POST /api/okr` | 所有 upsert 操作添加 `user_id: 'legacy'`（与 RLS policy 的 legacy 分支一致） |
| `GET /api/okr` | 添加 `.eq('user_id', 'legacy')` 过滤，不再返回所有用户混合数据 |
| DELETE diff 逻辑 | 仅查询当前用户记录对比，避免误删 |
| 后端错误处理 | 收集错误到数组，有错误时返回 500 + 详情，不再静默返回 success:true |
| 前端 save useEffect | 实际检查响应状态，失败时在标题下方显示红色错误提示，不再 `.catch(() => {})` |
| 数据库迁移 | 将 4 条旧记录的 user_id 从各异的 UUID 统一迁移为 `legacy` |

验证：静态检查 (lint + ts-check) 通过；GET/POST 冒烟测试通过；测试数据确认已写入数据库（user_id = 'legacy'）。
