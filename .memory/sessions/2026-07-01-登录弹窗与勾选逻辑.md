## Q4/A4（7/1）登录/未登录两种状态分别处理数据加载与保存

**用户原话**：
> 都不知道你到底改到哪里去了。我们是不是有一个没登录状态和登录状态下两个状态？啊，如果是这样的话。是不是分别要修改？

**AI 回答**：
确认了两种状态必须分别处理，之前没有区分，导致未登录时数据加载全走 API（返回 null），勾选状态和复盘内容都无法正确显示。

**未登录状态**：
- `reviewDays` 从 localStorage 扫描 `daily-review-*` keys（检查今日完成/明日待办是否有内容）
- `overrides` 从 localStorage 加载 `calendar-overrides-${year}`
- 点击切换勾选状态时保存到 localStorage
- 页面刷新时 500ms 内不拦截写入（不弹出登录框）

**登录状态**：
- `reviewDays` 从 API `/api/daily-review?action=list-days` 加载
- `overrides` 从数据库加载并保存
- 数据迁移：localStorage → 数据库

修改了 `year-calendar.tsx`：
1. reviewDays 加载：API 失败时从 localStorage 扫描 daily-review-* keys
2. overrides 加载：API 失败时从 localStorage 读取 calendar-overrides-${year}
3. toggleDay 保存：API 返回 null（未登录）时写入 localStorage
4. getDayStatus：有复盘内容优先显示 ✓（不受 overrides 影响）

## Q3/A3（7/1）勾选逻辑：有复盘内容才✓，没内容✗

**用户原话**：
> 那个我设置的那个就是每个日期嘛，我设置从一月开始嘛，它大多数日期都是打勾的，但是我点开之后，其实它的那个复盘里面并没有内容。那我之前的逻辑是有内容它才打勾，没内容就是差呀。这逻辑为什么都错了呢？

**AI 回答**：
根因：`getDayStatus` 函数中 `overrides[key]` 优先级最高，有 overrides 就直接返回，覆盖了基于 reviewDays 的判断。overrides（手动设置的勾/叉）变成了"覆盖"而非"兜底"。

修改 `getDayStatus` 函数优先级逻辑：
- **旧逻辑**：overrides > reviewDays > 默认
- **新逻辑**：有复盘内容 → 优先显示 ✓（不管 overrides）；没有复盘内容 → 看 overrides 或默认 ✗

核心变化：overrides 从"覆盖"变成"兜底"——只有没有复盘内容时才看 overrides。

## Q2/A2（7/1）刷新仍弹出登录框+try-catch语法错误

**用户原话**：
> 首先啊。我没登录的情况下，我刷新它还是会弹出这个登录，主动弹出。几次啊？那个我设置的那个就是每个日期嘛，它大多数日期都是打勾的，但是我点开之后，其实它的那个复盘里面并没有内容。那我之前的逻辑是有内容它才打勾，没内容就是差呀。这逻辑为什么都错了呢？

**AI 回答**：
这条消息包含两个问题：登录弹窗 + 勾选逻辑。

**登录弹窗部分**：前次修复（Q1）不够，_initialized 标记仍有时序问题——初始化期间其他组件写同步 key 仍会触发。添加 `_paused` 启动暂停期：拦截器安装后 500ms 内写入直接放行，不拦截也不弹窗。

**勾选逻辑部分**：见 Q3/A3。

**过程中犯错**：多次 edit_file 替换 year-calendar.tsx 同一个 try-catch 块，导致 try 块结构被破坏（缺少 catch/finally），TS 报错 TS1472。根因是对已多次修改的大文件用 edit_file 逐条替换，上下文不一致导致结构断裂。正确做法是先 read_file 确认当前状态，必要时整体重写该函数块。

## Q1/A1（7/1）登录弹窗：刷新时自动弹出，只有数据交互才应弹出

**用户原话**：
> 可是网页刷新的时候，它也会自动弹出，都不应该，只有在有数据交互的时候，它才应该弹出，需要登录才能才能数据交互

**AI 回答**：
根因分析：拦截器 `installAuthInterceptor()` 安装时，`_isLoggedIn` 默认 `false`，UserProvider 还在异步获取 session。期间其他组件初始化写入同步 key 就触发 `requireLogin()` 弹窗。

修复方案（auth-interceptor.ts）：
1. 拦截器安装时先获取 session 状态：`await supabase.auth.getSession()` 设置 `_isLoggedIn`
2. 添加 `_initialized` 标记：初始化期间不触发登录弹窗
3. `installAuthInterceptor` 改为 `async`，等待 session 检查完成

修改了 `sync-provider.tsx`：调用 `await mod.installAuthInterceptor()` 等待安装完成。

但此修复不够完整，500ms 后仍可能触发（见 Q2），最终加了 `_paused` 暂停期机制。
