# Cross Session Memory（跨对话记忆系统）

## 概述

解决 AI Agent 跨对话失忆问题。每个对话的上下文是割裂的，Agent 无法自动获取其他对话的决策、偏好和错误。本技能通过「索引 + 详情 + 错误集」三层结构，让 Agent 在新对话中精准回溯历史，定位到具体行号，不通读全文。

## 何时使用

- 新建对话开始时（必读 INDEX.md + PITFALL.md）
- 用户要求在新项目中部署记忆体系
- 对话结束前（必须更新详情 + 索引 + 错误集）
- 用户回老对话要求补存上下文时

## 核心结构

```
.memory/
├── INDEX.md              ← 索引文件（每条一行，指向详情文件的具体行号）
├── PITFALL.md            ← 错误合集（犯错场景 + 正确做法 + 根因）
└── sessions/
    └── YYYY-MM-DD-主题.md ← 对话详情（完整原文存档）
```

## 工作方式

### 启动时（每次新对话必做）

1. 读 `.memory/INDEX.md` → 扫一遍所有历史摘要
2. 读 `.memory/PITFALL.md` → 知道什么错不能犯
3. 需要某次对话细节？→ 按 INDEX 中的定位精准跳读
   - 例：`sessions/2026-06-30-周复盘.md#Q4(L61)` → `read_file(path, offset=61, limit=8)`

### 详情文件格式

```markdown
# 主题关键词 - YYYY-MM-DD

## 目录
| 编号 | 主题 | 行号 |
|------|------|------|
| Q1/A1 | 需求讨论 | L10 |
| Q2/A2 | 方案确认 | L22 |

## Q1/A1 需求讨论
用户问：（完整原话，一字不差）
AI答：（完整原话，一字不差）

## Q2/A2 方案确认
用户问：（完整原话）
AI答：（完整原话）
```

**关键规则**：
- 内容必须保留完整对话原文，不要概括
- 写完后必须回核对齐目录表中的行号
- 同一主题的延续对话，从上一个 Q 编号继续追加，禁止全量覆盖

### INDEX.md 格式

```markdown
# 对话索引

| 日期 | 主题 | 关键点 | 详情定位 |
|------|------|--------|----------|
| YYYY-MM-DD | 主题关键词 | 具体决策内容 | sessions/YYYY-MM-DD-主题.md#Q编号(L行号) |
```

**关键规则**：
- 一个对话如果涉及多个独立话题，拆成多条索引，每条指向不同 Q/A 轮次
- 关键点要写具体（"确认了XX方案"），不要写空泛（"讨论了XX"）
- 只在末尾追加，禁止全量覆盖
- 行号必须与详情文件目录表中的行号一致

### PITFALL.md 格式

```markdown
# 错误合集

## P01 错误标题

犯错场景：什么情况下犯了什么错
正确做法：① 具体步骤1 ② 具体步骤2
根因：为什么会犯这个错
```

**关键规则**：
- 每次犯错修完后立即追加，不遗漏
- 描述要具体到场景和步骤，不能笼统

### 对话结束时的更新流程

1. **更新详情文件**：追加新的 Q/A 轮次，更新目录表行号，禁止全量覆盖
2. **更新 INDEX.md**：在末尾追加新索引条目，禁止全量覆盖
3. **更新 PITFALL.md**：如有犯错，追加新条目
4. **git push**：推送后提醒用户"已推送到 GitHub"
5. **done**：交付

## 在新项目中部署

运行初始化脚本：

```bash
bash scripts/init-memory.sh
```

脚本会：
1. 创建 `.memory/` 目录结构
2. 生成空的 INDEX.md 和 PITFALL.md
3. 输出一段 AGENTS.md 配置，复制到项目的 AGENTS.md 中即可

## 注意事项

- 绝对禁止全量覆盖已有记忆文件，只追加
- 行号写完后必须回核对齐
- 用户的否定性表态（"我不要XX"、"XX无所谓"）必须保留原话
- git push 后必须提醒用户"已推送到 GitHub"，然后才能调用 done 交付
